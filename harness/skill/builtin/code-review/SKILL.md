# 代码审查指南

## 审查流程

1. **理解变更范围**：先浏览修改的文件列表，理解本次变更的目的
2. **逐文件审查**：按重要程度排序，核心逻辑优先
3. **交叉检查**：关注修改之间的一致性
4. **输出报告**：按严重程度分类给出反馈

## 审查清单

### 正确性
- [ ] 逻辑是否正确，是否覆盖了边界情况
- [ ] 错误处理是否完善（异常、空值、超时）
- [ ] 并发场景是否安全（如有）

### 安全性
- [ ] 是否存在注入风险（SQL 注入、命令注入、XSS）
- [ ] 敏感信息是否泄露（API Key、密码、token）
- [ ] 输入是否经过校验和清洗

### 可维护性
- [ ] 命名是否清晰、一致
- [ ] 函数/方法长度是否合理（建议 < 50 行）
- [ ] 是否有过度耦合或重复代码
- [ ] 注释是否充分（非显而易见的逻辑）

### 性能
- [ ] 是否存在 N+1 查询或不必要的循环
- [ ] 大数据集合操作是否高效
- [ ] 是否有不必要的内存分配

### 测试
- [ ] 是否有对应的单元测试
- [ ] 测试是否覆盖了关键路径和边界情况

## 反馈格式

按严重程度分为三级：

- **严重**（必须修复）：逻辑错误、安全漏洞、数据丢失风险
- **建议**（推荐改进）：代码风格、性能优化、可读性提升
- **可选**（锦上添花）：命名微调、注释补充

每条反馈包含：文件名 + 行号 + 问题描述 + 建议修改方案。
